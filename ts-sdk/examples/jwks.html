TN Protocol

JWKS Trust Demo

Simulate how a service verifies public encryption keys before sending vault-bound or project-to-project TN packages.

Real-World Workflow

A finance audit service needs to send encrypted nightly reports to a company vault.

  1. The audit service asks the vault for its public key set.
  2. An admin compares the vault issuer and fingerprint against the value shown in the dashboard.
  3. The project pins that exact issuer and fingerprint in `vault.jwks` and records `tn.jwks.pinned`.
  4. When the service sends a nightly report, it seals the payload to the vault's active encryption key.
  5. The report log records `tn.jwks.key_selected`, so an auditor can prove which vault key protected that report.
  6. If the vault rotates keys, the service records the previous fingerprint in `tn.jwks.rotated` before trusting the new set.
This simulation uses the local sample vault JWKS.

Project-to-Project Sharing

AuditCo publishes a public JWKS so a partner service can encrypt a handoff package back to AuditCo.

  1. AuditCo exports a public JWKS from its local TN project, including the JWE encryption public key.
  2. The partner receives the JWKS URL or file through an authenticated channel.
  3. The partner checks the issuer and fingerprint with AuditCo before trusting the key set.
  4. The partner seals a package to AuditCo's active encryption key and records the selected `kid`.
  5. AuditCo can decrypt with its local private `.jwe.mykey`; the private key never leaves AuditCo.
Uses the current JWKS input as AuditCo's published key set.
