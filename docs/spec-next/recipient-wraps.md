# Recipient Wraps

A recipient wrap is a sealed copy of a sealed-body BEK addressed to one
recipient by DID. The recipient unseals it with their Ed25519 seed
(converted to X25519 via the birational map), recovers the BEK, and
decrypts the package body. This is the sealed-box layer that sits on top
of the AES-GCM body frame in `docs/spec-next/body-encryption.md`.

## Wire Shape

Each entry in `manifest.state.body_encryption.recipient_wraps[]`:

```json
{
  "frame": "tn-sealed-box-v1",
  "recipient_identity": "did:key:z...",
  "ephemeral_x25519_pub_b64": "<standard base64, 32 bytes>",
  "wrap_nonce_b64": "<standard base64, 12 bytes>",
  "wrapped_bek_b64": "<standard base64, BEK ciphertext + 16-byte tag>"
}
```

All base64 fields use standard base64 with `=` padding (not URL-safe).
For one-recipient packages, producers also write the singular
`state.body_encryption.recipient_wrap` as a shadow of the sole array
entry.

## Algorithm

Seal, per recipient DID:

1. Recipient X25519 pub = `ed25519_pk_to_curve25519(did_ed25519_pub)`.
2. Ephemeral X25519 keypair `(eph_priv, eph_pub)`.
3. `shared = X25519(eph_priv, recipient_x_pub)`.
4. `wrap_key = HKDF-SHA256(shared, salt = eph_pub || recipient_x_pub,
   info = "tn-kit-seal-v1", len = 32)`.
5. `wrapped = AES-256-GCM(wrap_key, nonce=random(12), bek, aad)`.

Unseal recomputes the same `wrap_key` from the device seed and the
ephemeral pub, deriving the recipient X25519 pub from the wrap's
`recipient_identity` (not from the device seed) so a wrap naming a
different DID fails authentication rather than silently succeeding.

## AAD

The AES-GCM AAD binds the wrap to its manifest:

```text
aad = canonical_bytes(manifest minus {
  "manifest_signature_b64",
  "state.body_encryption.recipient_wrap",
  "state.body_encryption.recipient_wraps"
})
```

The signature is stripped because it is computed after the wraps; the
wrap blocks are stripped because they cannot bind themselves and because
each recipient only has their own view. Everything else — including the
rest of `state.body_encryption` — is bound, so a wrap cannot be lifted
into a different manifest.

## Constants

- frame: `tn-sealed-box-v1`
- HKDF info: `tn-kit-seal-v1`
- nonce: 12 bytes; ephemeral pub: 32 bytes; BEK: 32 bytes; tag: 16 bytes

## Implementations

The primitive lives in the Python and TS SDKs:

- Python: `python/tn/recipient_seal.py`
- TS: `ts-sdk/src/core/recipient_seal.ts`

Rust/WASM has no recipient-wrap implementation yet; Rust-only consumers
cannot unseal recipient-addressed packages. Moving the primitive into
shared core is consolidation work tracked in the audit matrix, out of
scope for the parity bar.

## Golden Vector

`tests/fixtures/recipient_wraps/vector.json` pins a deterministic
recipient seed, BEK, and representative manifest, plus a `wrap` produced
by the Python reference. The ephemeral key inside `wrap` is random per
seal, so the vector is an unseal golden: every implementation recomputes
the AAD from `manifest` and unseals `wrap` with `ed25519_seed_hex` to
recover `bek_hex`. Generated by
`python/tools/generate_recipient_wrap_fixture.py`; never hand-edited.

## Current Contract Tests

- Python: `python/tests/test_recipient_wrap_contract.py`
- TS: `ts-sdk/test/recipient_seal.test.ts`

Both load the shared vector, recompute the AAD, and unseal to the
expected BEK; both also cover the AAD strip list, seal/unseal round-trip,
outsider rejection, lift-attack rejection, and unknown-frame rejection.
