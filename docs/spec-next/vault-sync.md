# Vault Sync

Vault is the package-exchange backend for project continuity. The
communication unit is `.tnpkg` (see `docs/spec-next/tnpkg.md` and
`docs/spec-next/project-backup.md`): vault sync pushes/pulls packages and
auto-absorbs inbound ones additively (see `docs/spec-next/absorb.md`); it
never uploads application logs.

The live push/pull/auto-absorb loop talks to a running vault server and
is therefore **out of scope for the parity bar** here (CI excludes the
vault server tests). What this slice pins is the **server-independent**
half: how the project-level `vault:` config normalizes, which every SDK
must agree on before any network call.

## Vault Block Normalization

The runtime vault view is derived from the project-level `vault:` block,
with a legacy fallback to `ceremony.linked_vault` / `linked_project_id`:

- **No `vault:` block** → bridge from legacy `ceremony.linked_*`:
  `enabled` and `autosync` are true iff a legacy URL is present; interval
  defaults to 600.
- **`vault:` present** → `enabled` defaults to true; `autosync` defaults
  to `enabled`; `sync_interval_seconds` defaults to 600 and must be a
  positive integer; `enabled: true` requires a non-empty `url`.
- **`enabled: false`** → suppresses `url` / `linked_project_id` /
  `autosync` in the normalized view (the project is local-only) even if
  those fields are present.

Error conditions (raised consistently): `enabled: true` without `url`;
non-positive or non-integer `sync_interval_seconds`; a non-mapping
`vault:` value.

This rule is pinned across Python and TS by the shared golden
`tests/fixtures/vault/normalize_cases.json` (generated by
`python/tools/generate_vault_normalize_fixture.py`). Each side runs its
own normalizer (`_resolve_vault_settings` / `normalizeVaultConfig`) over
the `(vault, ceremony)` cases and asserts the normalized view, or that it
raises. Python additionally cross-checks against the Rust normalizer
(`tn_core.config_load_summary`) in the loaded-config path.

## Local Vault State Directory (forward target)

The intended local layout for package staging is
`.tn/<project>/vault/{pending,sent,received/{absorbed,failed},conflicts}`.
A shared planner for this lifecycle is not built yet; defining it (and the
unified `vault.sync` handler that drives it) is consolidation work tracked
in the audit matrix, not part of this parity slice.
