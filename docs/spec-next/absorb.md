# Absorb

Absorb is **additive capability intake**: verify a `.tnpkg`, then add any
usable keys, kits, contacts, log snapshots, or project state into the
target project. It is never a destructive import/restore. Absorb must
not replace the current project, replace an existing root identity,
rename the local project, or overwrite non-empty vault linkage.

## Flow

1. Open the `.tnpkg`, read `manifest.json`.
2. Verify the manifest signature before trusting body content (see
   `docs/spec-next/manifest.md`). Reject on failure.
3. If the manifest carries `state.body_encryption.recipient_wrap[s]`,
   unseal the body for this device before kind dispatch (see
   `docs/spec-next/recipient-wraps.md`). A package addressed only to
   other recipients is rejected as "not for me", not as tampered.
4. Dispatch on `manifest.kind`.

## Root Authority

Only a **root-authoritative** package may establish a new project root
identity in an empty location:

- `project_seed` and `identity_seed` are root-authoritative. They MUST be
  self-addressed (`publisher_identity == recipient_identity`), the body's
  `local.public` MUST equal `publisher_identity`, and the DID derived
  from `local.private` MUST match — a tamper guard that holds even though
  the manifest signature already verified.
- A `kit_bundle` / capability bundle MUST NOT silently become the root.
  Absorbing one with no project present mints a fresh local root identity
  and installs the kit as a capability under it.

Overwrite safety: a root-authoritative package whose `tn.yaml` differs
from an existing one is refused when the local log already has
user-emitted (non-`tn.*`) entries — overwriting would orphan signed
entries. A fresh ceremony (zero user events) may be replaced, with the
prior files renamed aside to `.previous.<UTC_TS>`.

## AbsorbReceipt

`tn.absorb(source)` returns an `AbsorbReceipt` (TS: the parity shape):

- `kind` — manifest discriminator that drove dispatch
- `accepted_count` / `deduped_count` — units applied vs already present
- `noop` — receiver clock dominates; nothing to do
- `conflicts` — equivocation signals (e.g. leaf-reuse after revoke)
- `replaced_kit_paths` — local files renamed aside to make room
- `derived_state` — for admin snapshots, the reduced state

## Vault Metadata Adoption

When a root-authoritative `project_seed` is absorbed over an existing
project YAML, vault metadata is filled **only where the local value is
empty**, and never overwritten:

- empty local `vault.url` / `vault.linked_project_id` → filled from the
  package; `enabled: true` and default `autosync` / `sync_interval_seconds`
  are set when anything was adopted.
- non-empty local values → preserved; the package's values are ignored.
- `vault.enabled: false` locally → adoption is blocked entirely.
- when the only difference is non-adoptable vault metadata, the YAML is
  treated as deduped (`vault_only`), not a conflicting overwrite.

This rule is pinned across Python and TS by the shared golden
`tests/fixtures/absorb/vault_adoption_cases.json` (generated by
`python/tools/generate_absorb_vault_adoption_fixture.py`). The two SDKs
serialize YAML differently, so the golden records the semantic outcome
(changed?, vault_only?, resulting vault/ceremony values), and each side
runs its own patch (`_project_seed_vault_yaml_patch` /
`_projectSeedVaultYamlPatch`) and asserts the parsed result matches.
