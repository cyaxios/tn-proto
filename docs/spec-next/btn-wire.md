# BTN Wire

The BTN binary wire format — ciphertext, reader-kit, and publisher-state
layouts — is **owned by Rust** (`crypto/tn-btn`). No SDK reimplements it:
Python reaches it through the `tn_btn` PyO3 binding, and TS/browser reach
it through WASM (`btn_decrypt_js`, re-exported as `btnDecrypt`). This is
the one surface that is legitimately Rust-single-source already; the
parity bar here is proving every language decodes the *same* bytes, not
porting anything.

## Wire Header

Every BTN wire record begins with a 3-byte header:

```text
byte 0: magic   0xB7
byte 1: version 0x01
byte 2: kind    0x01 = ciphertext
                0x02 = reader kit
                0x03 = publisher state
```

The rest of each record (publisher id, epoch, leaf entries, nonces,
lengths, body) is the binary layout defined in `crypto/tn-btn/src/wire.rs`.

## Shared Golden

`crypto/tn-core/tests/fixtures/btn_vectors.json` (generated by
`python/tools/generate_rust_fixtures.py::gen_btn`) carries a publisher
seed/state, a reader kit, and a ciphertext over the plaintext
`hello btn`. Every implementation decodes the same reader kit +
ciphertext to the same plaintext:

- Rust: `crypto/tn-core/tests/cipher_btn.rs`
- Python: `python/tests/test_btn_wire_contract.py` (`tn_btn.decrypt`)
- TS/WASM: `ts-sdk/test/btn_wire_contract.test.ts` (`btnDecrypt`)

The Python and TS tests additionally assert the magic/version/kind header
of each record, so a layout drift is caught before decrypt, and the
Python test round-trips the publisher-state blob through
`tn_btn.PublisherState.from_bytes` / `to_bytes`.

## Out of Scope (Parity Bar)

Exposing a full BTN parser/serializer surface (beyond decrypt) through
Python and TS, or moving wire handling out of `crypto/tn-btn`, is
consolidation, not parity. The slice here is the cross-language decode
golden + header pin.
