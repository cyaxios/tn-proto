### [P1] _AdvisoryFileLock.__enter__ leaks the per-path process lock (and fd) on failure — later keystore writes deadlock the process
- file: python/tn/_keystore_backend.py:135  (lens: concurrency, concurrency / locking)
- desc: __enter__ acquires the per-path threading.Lock first (line 135), then performs operations that can raise: os.open of the lock file (line 141) and, on Windows, msvcrt.locking with LK_LOCK (line 163) which the code's own comment (lines 156-160) documents as raising OSError after ~10 retries at ~1s intervals. When __enter__ raises, Python never calls __exit__, so the threading.Lock in the module-level _PROCESS_LOCKS registry stays held forever (and on the msvcrt path the open fd also leaks, keeping the lock file pinned). Every subsequent write_state() for that group in the same process then blocks indefinitely at self._proc_lock.acquire() — not an error, a permanent hang. The cross-process OS lock is held by another process legitimately for >10s and is released, but this process is bricked. Fix is to wrap the body after acquire in try/except that releases the proc lock and closes the fd before re-raising (or acquire the OS lock before the thread lock).
- scenario: Windows (the SDK's supported CLI+library concurrency): an app process calls tn.admin.add_recipient (BtnGroupCipher._persist_state -> write_state) while a scheduled 'tn rotate' CLI run, an antivirus scan, or a suspended debugger holds the .btn.state.lock file for more than ~10 seconds. msvcrt.locking raises OSError; the admin verb fails — acceptable — but the per-path threading.Lock is never released, so the NEXT keystore write in the app (any add/revoke/persist for that group) blocks forever. The application thread hangs with no timeout and no diagnostic.
- evidence: python/tn/_keystore_backend.py:134-141: '# In-process serialisation first. Released last on exit.\nself._proc_lock.acquire()\nself._proc_lock_held = True\n...\nself._fd = _os.open(self._path, flags, 0o600)' and 161-163: 'import msvcrt as _msvcrt\n\n_msvcrt.locking(self._fd, _msvcrt.LK_LOCK, 1)' — both after the acquire, with release only in __exit__ (lines 189-192), which never runs when __enter__ raises.
